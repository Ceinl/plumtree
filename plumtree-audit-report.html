PT-01

Build work happens before authorization, validation and quota enforcement

Resolved

Revalidated: requests are now validated, update claims authenticated, and deploy quota reserved before building. Artifact and deploy creation is atomic, with build concurrency and queue limits covered by tests.

A deploy or update is compiled and persisted as an artifact before the deploy claim is validated. On update, possession of the claim token is not checked until after the build. Invalid names, bad deploy IDs, bad claim tokens and exceeded deploy quotas therefore still consume a build and leave an unreferenced artifact that has no cleanup path.

ImpactAnyone holding the published dev token can repeatedly consume CPU, memory and durable blob storage. A random bearer token is enough on the update route. The configured deploy/hour cap does not protect the expensive operation.
FixValidate request fields, authenticate the deploy claim and reserve quota before calling the builder. Create artifact + deploy atomically, or delete the artifact on every failure. Add a build concurrency semaphore.

Evidence: dev_deploy.go:43–68, dev_deploy.go:147–166, dev_deploy_artifact.go:28–64

PT-02

Standalone gateway truncates normal WASM responses

Resolved

Revalidated: artifact responses now use a separate 256 MiB limit, while metadata stays at 1 MiB. An HTTP-backend test successfully round-trips a 3 MiB artifact.

The gateway applies a 1 MiB LimitReader to every successful backend response, including ResolveRunnable, whose JSON contains base64 WASM. The scaffolded counter builds to 2,741,026 bytes (about 3,654,704 base64 bytes), so production resolution ends with truncated JSON.

ImpactOrdinary deployed Go apps fail to launch through the standalone gateway used by Docker Compose.
FixStream artifacts separately, or use an explicit artifact-size cap large enough for the response. Keep the 1 MiB limit only for small metadata/error responses and add an adapter integration test with a multi-megabyte artifact.

Evidence: httpbackend/client.go:35, httpbackend/client.go:129

PT-03

CLI apps bypass worker-process isolation

Resolved

Revalidated: the process protocol supports CLI arguments and filtered output, and the gateway routes CLI apps through ProcessRunner.RunCLI when a worker is configured.

The production image configures RunnerWorker, but the gateway sends CLI apps directly to the in-process Runner.RunCLI. Only TUI sessions use ProcessRunner.

ImpactA runtime or host-function escape in a hostile CLI app lands inside the public SSH gateway process, contrary to the documented per-session process boundary.
FixImplement CLI mode in the process protocol and route both app types through the worker whenever it is configured. Add a production-mode test asserting the worker is spawned.

Evidence: gateway/session.go:122–157, procworker.go:28–34

PT-04

SSH clients control unbounded gateway allocations

Resolved

Revalidated: PTY and resize requests are rejected outside 1–500 × 1–300 with a 150,000-cell cap; the screen allocation site independently enforces the same bounds.

PTY and window dimensions are accepted as uint32 with no bounds. Before the isolated worker starts, the gateway constructs a screen with h × w cells in its own process.

ImpactAn anonymous SSH client can request extreme dimensions and trigger excessive allocation or process OOM, taking down all tenants on the gateway.
FixClamp dimensions at request parsing (for example 1–500 columns and 1–300 rows), reject invalid requests, and cap total cells before every screen resize.

Evidence: gateway/session.go:44–57, gateway/session.go:134–147, screen.go:30–51

PT-05

SSH connections have no admission or idle limits

Resolved

Revalidated: global/per-IP admission, handshake and activity-based idle deadlines now have secure defaults. Session capacity is reserved before resolving and retaining WASM.

Each accepted TCP connection gets a goroutine; no handshake deadline, idle deadline, connection cap or per-IP cap is applied. After handshake, the complete WASM artifact is resolved and retained before the client opens a session channel. The session semaphore is acquired later.

ImpactSlow handshakes exhaust file descriptors/goroutines; many idle authenticated connections retain multi-megabyte artifacts and bypass MaxConcurrentSessions.
FixAdd handshake and idle deadlines, global/per-IP connection admission, and acquire capacity before artifact resolution. Avoid embedding artifact bytes in per-connection state.

Evidence: gateway/server.go:101–152, gateway/session.go:79–89

PT-06

Production abuse limits default to unlimited

Resolved

Revalidated: control plane, gateway and worker ship nonzero defaults; production mode rejects unlimited critical settings unless explicitly acknowledged. Compose adds CPU, memory and PID limits.

Global concurrent sessions and deploys/hour both default to zero (unlimited); Compose does not set either. The build service also has no job concurrency limit, and the container has no memory limit. The dev token is intentionally recoverable from released binaries.

ImpactA public deployment starts with weak protection against build floods and cross-app session floods. Multiple 2 GiB-limited compiler processes can still OOM the host.
FixShip conservative nonzero defaults, configure Compose limits, add build queue/concurrency controls and container CPU/memory/PID limits. Refuse production startup when critical limits remain unlimited unless explicitly acknowledged.

Evidence: control-plane main.go:87–90, ssh-gateway main.go:78, docker-compose.yml

PT-07

Suspension does not terminate live sessions

Resolved

Revalidated: suspensions fan out to embedded and standalone gateways, cancel matching sessions, wait for deregistration, and acknowledge completion before the operation returns.

Store suspension flags block future resolution only. A session registry can cancel active sessions, but no operator path calls it; the limitation is explicitly marked TODO.

ImpactMalicious or compromised code continues running until disconnect or the 30-minute session timeout after an operator “suspends” it.
FixWire owner/app/deploy suspension events to every gateway and wait for cancellation acknowledgement.

Evidence: killswitch.go:29–34, store_deploys.go:270–316

PT-08

Public HTTP servers omit defensive timeouts

Resolved

Revalidated: control plane and build worker now set read-header, read, write and idle timeouts plus bounded graceful shutdown.

The control plane and build worker instantiate http.Server with only address and handler. Read-header, read, write and idle timeouts are unset.

ImpactDirect exposure permits slow-request connection exhaustion. A reverse proxy may reduce but does not remove the application-layer weakness.
FixSet explicit ReadHeaderTimeout, ReadTimeout, WriteTimeout, IdleTimeout and bounded shutdown contexts.

Evidence: control-plane main.go:178, build-worker main.go:44

PT-09

No user-facing SSH key enrollment path

Low

Revalidated and downgraded: “none” authentication is gone; the gateway records a fingerprint only after signature verification, distinguishes anonymous/unregistered/registered keys, and resolves registered keys through the control plane.

The remaining gap is product plumbing: no HTTP or CLI route lets a normal owner call the existing RegisterSSHKey operation, so owner-linked Authenticated=true identities require out-of-band provisioning.

ImpactStable proved-key identities work, but self-service owner association is unavailable.
FixAdd authenticated owner endpoints and pt commands to list, register and revoke SSH keys.

Evidence: gateway/server.go, store_credentials.go; no enrollment route found under control-plane/internal/httpapi or pt.

PT-10

Issued CI tokens cannot authorize any operation

Resolved

Revalidated: the incomplete HTTP token-issuance API and routes were removed, so users are no longer issued credentials that appear usable but are not.

The API creates scoped ptci_ tokens and returns their plaintext once, but no HTTP route calls AuthenticateCIToken or enforces token scopes. Deploy, inspect, logs and secrets still require the global dev token plus claim token.

ImpactUsers can create credentials that appear valid but are unusable, and the intended replacement for shared deploy credentials is incomplete.
FixAdd CI bearer authentication middleware and scope checks to each operation, or remove the token UI/API until complete.

Evidence: tokens.go:19–72, store_credentials.go:126–142

PT-11

Dashboard bearer token is placed in the URL

Resolved

Revalidated: the query-token fallback was removed. The dashboard now consumes the SSE response through fetch with an Authorization header and reconnect handling.

The SSE endpoint accepts access_token and the dashboard constructs an EventSource URL containing the Shoo bearer token.

ImpactTokens can enter access logs, proxy telemetry, browser history and diagnostic captures. Referrer-Policy helps only with referrer disclosure.
FixUse a same-origin, short-lived, single-purpose SSE ticket or replace EventSource with a fetch-based stream that carries an Authorization header.

Evidence: http_helpers.go:90–95, templates.go:160–166

PT-12

File-backed app KV retains mutations after persistence failure

Resolved

Revalidated with forced failure: FileStore now serializes mutation and persistence, restores the previous value and byte count on error, and has explicit write- and rename-failure regression tests. A forced parent-directory failure returned an error and the attempted key was not visible afterward.

The original implementation mutated the embedded memory store before writing the JSON file. A mkdir, write or rename failure returned an error while leaving the rejected value visible to live sessions.

Original impactApp KV state could diverge from its acknowledged durable state and disappear after gateway restart.
Implemented fixMutation, snapshot and rollback are guarded by a transaction lock; failed persistence restores both data and quota accounting.

Evidence: runner/kv.go: FileStore.Set/Delete

PT-13

Application secrets are stored as reversible plaintext data

Resolved

Revalidated: persistent snapshots now use AES-GCM envelope encryption with a KEK mounted outside the state volume, support key rotation, and production mode refuses persistent startup without the key.

Secret bytes are base64-encoded into the JSON state snapshot. File mode limits casual access, but volume snapshots and backups contain immediately recoverable values and there is no encryption or key separation.

ImpactA leaked control-plane volume or backup exposes every tenant secret.
FixUse a managed secret store or envelope encryption with a key held outside the data volume; document rotation and backup handling.

Evidence: persistence_snapshot.go:52–58, persistence.go:28–45

PT-14

Rate limiting can permanently hide the final TUI frame

Low

Frames denied by the sink throttle update the in-memory screen but schedule no later flush; the dirty field is unused. A last frame arriving inside the interval may never reach the terminal.

ImpactUsers can see stale state or miss a completion/error screen until another event causes a later render.
FixCoalesce to the latest frame and schedule a trailing-edge flush; flush pending content on shutdown.

Evidence: render.go:48–90

PT-15

Production adapters and entrypoints lack direct tests

Low

ssh-gateway/httpbackend, both standalone command packages and several SDK example paths report “no test files.” The 1 MiB production gateway failure is therefore outside current coverage even though all module tests pass.

ImpactConfiguration-specific failures survive a green suite.
FixAdd Docker-topology smoke tests: deploy a real scaffolded app, resolve it through the HTTP backend, run both CLI and TUI sessions, exercise suspension and hostile SSH dimensions.
